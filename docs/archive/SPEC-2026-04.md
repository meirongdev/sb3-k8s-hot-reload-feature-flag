> **约束**：
> - 排除 **Spring Cloud Config Server**（中心化 git-backed 配置服务器）
> - 排除 **Netflix 套件**（Eureka / Zuul / Ribbon / Hystrix）
> 
> **明确允许**：
> - `spring-cloud-context`（提供 `@RefreshScope`、`EnvironmentChangeEvent`、`/actuator/refresh` 的 bean 刷新效果）
> - `spring-cloud-kubernetes-client-config` / `spring-cloud-kubernetes-fabric8-config`（直接读 K8s API 的 ConfigMap/Secret，可替代 Config Server）
> - `spring-cloud-kubernetes-configuration-watcher`（独立 controller pod，监听 ConfigMap 变化后触发应用刷新）
> - `spring-cloud-bus`（如有大规模 fan-out 需要）
> 
> **目标**：在 kind 集群上验证各种热加载方案的可行性、延迟、风险
> 
> **版本说明**：约束指定 Spring Boot 3.5。**Spring Boot 4.0 已于 2025-11 GA**，建议同时跑通 4.0 路径作为升级评估。Spring Boot 4 的 `BeanRegistrar`、Spring Data AOT、模块化 jar、resilience 注解（`@Retryable`/`@ConcurrencyLimit`）对本 spec 不构成阻碍，但需在 § 12 验证兼容性。

## 1. 背景与问题定义

### 1.1 热加载的驱动场景

| 场景 | 变更类型 | 频率 |
|------|---------|------|
| 动态调整日志级别（排查 prod 问题） | 配置 | 高频 |
| 调整限流阈值 / 超时时间 | 配置 | 中频 |
| 开关某个实验性功能（feature flag） | 配置/逻辑 | 中频 |
| 调整线程池大小 / 连接池参数 | 配置 | 低频 |
| 业务代码变更 | 字节码 | 中频 |
| 数据库 schema 变更 | DDL | 低频 |

### 1.2 核心问题

约束排除的是 **中心化 Config Server** 和 **Netflix 套件**，不是整个 Spring Cloud 生态。`spring-cloud-context` 提供的 `@RefreshScope`、`EnvironmentChangeEvent`、`/actuator/refresh` bean 刷新等核心机制**可用**，主流方案直接适用。

需要评估的真正决策点：
1. 用 `spring-cloud-kubernetes-*-config` 直接读 K8s API，还是 ConfigMap volume mount + `configtree`
2. 用 `spring-cloud-kubernetes-configuration-watcher`（独立 controller）触发刷新，还是应用内置 PropertySource Reload（**官方已 deprecated since 2020**），还是干脆走 Stakater Reloader 滚动重启
3. 哪些场景值得 in-process 刷新，哪些应回归滚动重启

## 2. 热加载 vs 滚动更新：Tradeoff 分析

### 2.1 决策树

```
变更是否涉及字节码？
  └─ 是 → 必须滚动更新（或 HotSwap，但限制极大）
  └─ 否 → 变更是否涉及安全敏感数据（credentials/keys）？
            └─ 是 → 优先滚动更新（避免内存中明文驻留过久）
            └─ 否 → 变更是否要求严格原子性（多个 pod 同时生效）？
                      └─ 是 → 滚动更新（ConfigMap 文件同步有时间差）
                      └─ 否 → 热加载是合理选项
```

### 2.2 Tradeoff 矩阵

| 维度 | 热加载 | 滚动更新 |
|------|--------|---------|
| 生效延迟 | 秒级（取决于轮询/watch 间隔） | 分钟级（镜像构建 + 拉取 + 健康检查） |
| 流量中断 | 无 | 无（readiness probe 保护下） |
| 一致性 | 弱（各 pod 生效时间不同） | 强（rolling 策略控制） |
| 状态安全 | 高风险（in-flight 请求中途看到新配置） | 低风险（老 pod 处理完才退出） |
| 可回滚性 | 需手动再次修改 ConfigMap | `kubectl rollout undo` 一键回滚 |
| 可观测性 | 需额外记录"何时谁加载了什么" | git/image tag 天然作为 audit trail |
| 调试难度 | 高（pod 之间配置版本可能不同） | 低 |
| 适合变更类型 | 运行时可调参数（log level, flag, threshold） | 所有其他变更 |

### 2.3 推荐原则

- **默认选择滚动更新**：可预测、可审计、与 GitOps 工作流天然契合。
- **热加载仅用于**：需要在不重启情况下快速调整的运行时参数，且该参数的短暂不一致性业务可以接受。
- **绝对不热加载**：数据库凭证、TLS 证书、启动 port、JVM 参数。

## 3. 技术方案清单

> 本节按推荐度排列。方案 A 是 Spring I/O 2025 大会主流推荐路径，与本 spec 约束兼容。

### 方案 A（**推荐**）：Spring Cloud Kubernetes Configuration Watcher + spring-cloud-context

**原理**

1. **配置读取**：应用启动时通过 `spring-cloud-kubernetes-fabric8-config`（或 `*-client-config`）直接从 K8s API 读取 ConfigMap/Secret，注入为 PropertySource。**或**走 ConfigMap volume mount + `spring.config.import=configtree:/config/`，二选一。
2. **变更监听**：在集群中部署一个独立的 **`spring-cloud-kubernetes-configuration-watcher`** controller pod。它监听所有带 `spring.cloud.kubernetes.config=true` 标签的 ConfigMap/Secret。
3. **刷新触发**：ConfigMap 变更后，Watcher 通过 HTTP 直接 `POST /actuator/refresh` 到所有匹配的应用 pod（或通过 Spring Cloud Bus 广播 `RefreshRemoteApplicationEvent`）。
4. **Bean 刷新效果**（来自 `spring-cloud-context`）：
   - `@ConfigurationProperties` beans 自动 rebind
   - `@RefreshScope` beans 在下次访问时用新配置重建（lazy proxy）
   - log levels 自动更新
5. **不要使用** Spring Cloud Kubernetes 内置的 PropertySource Reload 模式（`spring.cloud.kubernetes.reload.enabled=true`）——**官方自 2020 起 deprecated**，Joris Kuipers 在 Spring I/O 2025 明确不推荐。

**关键依赖**

```xml
<!-- 应用侧 -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-kubernetes-fabric8-config</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
<!-- 不需要 spring-cloud-config（被排除），不需要 spring-cloud-netflix（被排除） -->
```

部署 Configuration Watcher：

```bash
# 官方镜像
kubectl apply -f https://raw.githubusercontent.com/spring-cloud/spring-cloud-kubernetes/main/spring-cloud-kubernetes-controllers-configuration-watcher/...
```

**ConfigMap 标签要求**：

```yaml
metadata:
  name: demo-config
  labels:
    spring.cloud.kubernetes.config: "true"
data:
  application.yaml: |
    app:
      rate-limit: 100
```

**优点**：
- 与 Spring 大会主流推荐一致，社区案例多
- 应用代码标准（`@RefreshScope` / `@ConfigurationProperties`），无需自实现 Watcher
- 触发延迟可控（Watcher 是 event-driven，毫秒级感知）

**缺点**：
- 需要在集群中多部署一个 controller pod
- **GraalVM native image 不支持**（`@RefreshScope` 在 AOT 编译下失效）

---

### 方案 B：Spring Cloud Kubernetes 应用内 PropertySource Reload（**不推荐**）

设置 `spring.cloud.kubernetes.reload.enabled=true`，应用自身通过 K8s API watch 或 polling 监听 ConfigMap 变化。

**为什么不用**：
- 官方文档标注 deprecated since 2020
- 每个 pod 独立 watch K8s API，副本多时对 API server 有压力
- Joris Kuipers 在 Spring I/O 2025 明确说"用 Configuration Watcher 替代它"

**仅在以下场景考虑**：副本数极少（≤2）、不愿引入额外 controller、且团队充分理解其局限。

---

### 方案 C：Actuator `/actuator/refresh` + 自建外部触发

**原理**：与方案 A 类似，但用自建脚本/sidecar 替代 Configuration Watcher。

**触发器选项**：
- **Argo CD Webhook**：ConfigMap 变更同步完成后 webhook 通知应用 pod
- **inotifywait Sidecar**：监听挂载的 ConfigMap 文件变更，curl `/actuator/refresh`
- **CronJob**：低频场景下定时 refresh

**适用场景**：已经在用 ArgoCD/Flux，希望刷新触发逻辑由 GitOps 工作流完成，而不是引入 Spring Cloud Configuration Watcher controller。

**注意**：依然需要 `spring-cloud-context` 才能让 `/actuator/refresh` 真正刷新 `@RefreshScope` beans 和 rebind `@ConfigurationProperties`。纯 spring-boot-actuator 的 `/refresh` 只能改 logger，不能刷新 bean。

---

### 方案 D：纯 Spring Boot 自实现 WatchService + 手写 Rebinder（**仅理论选项**）

完全不引入 spring-cloud-context，用 Java `WatchService` 监听 mount 的 ConfigMap 文件，自己写 `ConfigurableEnvironment` 更新逻辑、自己反射 rebind `@ConfigurationProperties` beans。

**何时考虑**：
- 团队有特殊原因要彻底剥离所有 spring-cloud-* 依赖（例如 GraalVM native image 兼容性，或极致最小依赖洁癖）
- 准备承担"自己维护 rebind 边缘 case"的工程债务

**风险**：方案 A 已经把这套逻辑做对了 10 年，自实现极易踩坑（partial update、binding error 时的状态、scoped bean 生命周期、嵌套 ConfigurationProperties）。**不建议作为默认路径**。

---

### 方案 E：JVM HotSwap / DCEVM（仅开发环境）

**适用范围**：本地开发，不适合 production  
**实现**：JRebel 或 HotswapAgent（开源）+ DCEVM  
**Java 25**：需确认 DCEVM 是否支持最新 JDK（通常滞后 1-2 个大版本）

---

### 方案 F：Stakater Reloader 滚动更新（基准对照组，GitOps 推荐）

```yaml
metadata:
  annotations:
    reloader.stakater.com/auto: "true"
```

部署 Reloader：

```bash
helm install reloader stakater/reloader \
  --set reloader.reloadStrategy=annotations  # GitOps 推荐
```

**特点**：
- 不是真的"热加载"，是 ConfigMap 变更触发滚动重启
- 业界 K8s 社区无人质疑的默认答案
- 与 ArgoCD/Flux 兼容（用 annotation 策略）

结合：
- **Jib** 构建镜像（无需本地 Docker，增量层缓存）
- **kind load docker-image** 将镜像加载到 kind 节点

**目标**：从 ConfigMap 变更到所有 pod 跑新配置 < 3 分钟。

```yaml
  - role: worker
```

### 4.2 测试应用

```
demo-app/
├── src/main/java/
│   └── demo/
│       ├── config/
│       │   ├── AppConfig.java          # @ConfigurationProperties，标注 @RefreshScope（仅方案 D 自实现 rebind）
│       │   └── RefreshableBean.java    # @RefreshScope bean，验证刷新生效
│       ├── controller/
│       │   └── ConfigController.java   # 返回当前配置值
│       └── DemoApplication.java
├── k8s/
│   ├── configmap.yaml                  # 带 spring.cloud.kubernetes.config=true 标签
│   ├── deployment.yaml
│   ├── service.yaml
│   └── configuration-watcher.yaml      # 方案 A 用
└── pom.xml
```

关键端点：

```
GET /config             # 返回当前所有 config 值（用于验证热加载是否生效）
GET /actuator/health    # 含 RefreshHealthIndicator
GET /actuator/configprops  # Joris 推荐用此端点验证 rebind 结果
POST /actuator/refresh  # 由 Configuration Watcher 调用（方案 A），或手动（方案 C）
```

### 4.3 验证指标

| 指标 | 热加载目标 | 滚动更新基线 |
|------|-----------|------------|
| 配置生效延迟（end-to-end） | < 30s | < 180s |
| 生效期间错误率 | 0% | 0%（readiness 保护） |
| 多 Pod 配置一致性窗口 | < 5s（Configuration Watcher 是 event-driven） | 0s（不存在中间态） |
| 内存泄漏（连续 reload 10 次） | 无增长 | N/A |
| 热加载失败后是否降级 | 保留旧配置，`Refresh` HealthIndicator 报 DOWN | N/A |
| GraalVM native 兼容性 smoke test | 启动成功；refresh 时记录预期失效行为 | 启动成功 |

### 4.4 测试用例

**TC-01: 正常热加载路径**
1. 部署应用，`GET /config` 验证初始值
2. `kubectl edit configmap demo-config` 修改某个值
3. 每 5s 轮询 `GET /config`，记录生效时间
4. 断言：所有 Pod 在 T+60s 内都返回新值

**TC-02: 热加载期间的并发请求**
1. `hey -c 50 -z 120s http://<svc>/echo` 持续压测
2. 并发修改 ConfigMap 3次（间隔 20s）
3. 断言：无 5xx 响应，响应时间 P99 < 500ms

**TC-03: 无效配置注入**
1. 注入一个非法格式的配置值
2. 断言：应用保留上一次有效配置，不崩溃，告警日志出现

**TC-04: GraalVM native image 兼容性 smoke test**
1. 用 `mvn -Pnative native:compile` 构建 native image
2. 部署后修改 ConfigMap，验证 refresh 行为
3. 断言：启动成功；记录 `@RefreshScope` / rebind 在 native 下的实际行为（预期：失效或部分失效）。决定是否给 native 路径单独留一条"必须滚动重启"的 ADR

**TC-05: 滚动更新基线测试（方案 F）**
1. 修改 ConfigMap，让 Reloader 触发滚动
2. `hey -c 50 -z 120s http://<svc>/echo` 压测期间观察
3. 记录端到端时间和错误率

**TC-06: 大规模副本一致性（3 Pod）**
1. 部署 3 副本
2. 修改 ConfigMap
3. 每 2s 分别 curl 3 个 Pod 的 `/config`（通过 `kubectl port-forward` 绕过 LB）
4. 记录各 Pod 生效时间差（方案 A 期望 < 5s，方案 D 自实现期望 < 30s）

## 5. 核心实现：方案 A 参考代码

### 5.1 ConfigurationProperties + RefreshScope

```java
@ConfigurationProperties(prefix = "app")
@Component
public class AppConfig {
    private int rateLimit = 100;
    private Duration timeout = Duration.ofSeconds(5);
    // getters/setters — 业务代码必须读取 bean 的 property，不要直接读字段
}
```

```java
// 需要在变更时整体替换的 bean（持有连接池、客户端等场景）
@Component
@RefreshScope
public class RateLimiter {
    public RateLimiter(AppConfig config) {
        // 构造时拿到当前快照
    }
}
```

### 5.2 application.yml

```yaml
spring:
  application:
    name: demo-app
  cloud:
    kubernetes:
      config:
        sources:
          - name: demo-config
            namespace: default
        # 不要启用 reload（已 deprecated），改用 Configuration Watcher
        reload:
          enabled: false

management:
  endpoints:
    web:
      exposure:
        include: health, info, refresh, env, configprops, loggers
  endpoint:
    refresh:
      access: read-write
    env:
      access: read-write   # 仅在内网/受控环境开放，POST /env 可改单个属性
  health:
    refresh:
      enabled: true        # 暴露 RefreshHealthIndicator，bind 失败时报 DOWN
```

### 5.3 ConfigMap 与 Watcher 部署

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: demo-config
  labels:
    spring.cloud.kubernetes.config: "true"   # 必须，Configuration Watcher 据此筛选
data:
  application.yaml: |
    app:
      rate-limit: 100
      timeout: 5s
    logging:
      level:
        demo: INFO
```

```yaml
# 应用 Deployment 关键 annotation（让 Watcher 知道刷新方式）
metadata:
  annotations:
    spring.cloud.kubernetes.configuration.watcher.refreshStrategy: "http"  # 或 "bus"
```

部署 Configuration Watcher（独立 controller pod）：

```bash
kubectl apply -f k8s/configuration-watcher.yaml
# 或用官方 helm chart（社区维护版本）
```

### 5.4 ServiceAccount RBAC

应用 pod 需要从 K8s API 读 ConfigMap，Watcher 需要 watch ConfigMap：

```yaml

# Watcher：list/watch configmaps + secrets，POST 到应用 actuator
# 详见 spring-cloud-kubernetes 官方 RBAC 模板
```

### 5.5 验证 refresh 是否生效


### 6.2 后端选项（按复杂度排序）

| 后端 | K8s 原生 | 无额外服务 | 热加载 | 适合阶段 |
|------|---------|-----------|--------|---------|
| **flagd + ConfigMap** | ✅ | ✅（独立 Deployment / sidecar / in-process 三选一，详见 § 6.3 + § 12.3） | ✅ | MVP |
| **Flagsmith** | 需部署 | ❌ | ✅ | 成长期 |

| **LaunchDarkly** | SaaS | ❌ | ✅ | 企业级 |
| **自建（ConfigMap + 本 spec 方案 A）** | ✅ | ✅ | ✅ | 极简 MVP |

### 6.3 flagd + ConfigMap 方案（最小可行）

flagd 在 K8s 上有三种拓扑，**默认推荐 centralized**（本 demo 实际选用），见 § 12.3 演进路径：

```
                centralized（默认 / 本 demo）
┌──────────────┐        ┌──────────────┐
│  gateway pod │──gRPC──►│              │
└──────────────┘        │ flagd        │      ┌──────────────┐
┌──────────────┐        │ Deployment   │◄─────│  ConfigMap   │
│  ui (nginx)  │──HTTP──►│ (1 replica)  │ watch│ (flags.json) │
│  /ofrep/*    │  OFREP  └──────────────┘      └──────────────┘
└──────────────┘

                sidecar（每 pod 一份 flagd）
┌────────────────────────────────────────┐
│                  Pod                   │
│  ┌─────────────┐    ┌────────────────┐ │
│  │  App        │◄──►│  flagd sidecar │ │  从 ConfigMap 读规则
│  └─────────────┘    └────────────────┘ │
└────────────────────────────────────────┘

                in-process（sidecarless，2026 推荐升级版）
┌────────────────────────────┐    ┌──────────────┐
│  App + flagd evaluator     │◄───│ central flagd│ sync 协议推规则
│  (在 JVM 内评估，无 RPC)    │    │ (规则源)     │
└────────────────────────────┘    └──────────────┘
```

| 拓扑 | RPC 跳数 | 资源占用 | 何时选 |
|---|---|---|---|
| centralized（**本 demo**） | 1 跳 gRPC / HTTP | 1 个 flagd pod | MVP / 单集群 / 评估延迟 ms 级可接受 |
| sidecar | 0 跳（localhost） | N 个副本 = N 个 flagd 容器 | 应用对 flag 评估强延迟敏感、且不愿走 in-process |
| in-process | 0 RPC（JVM 内） | 1 个中心 flagd（仅推规则） | 副本数大、想去 sidecar、Java provider 0.13+ 已支持 |

```json
// ConfigMap: flags.json
{
  "flags": {
    "new-checkout-flow": {
      "state": "ENABLED",
      "variants": { "on": true, "off": false },
      "defaultVariant": "off",
      "targeting": {
        "fractional": [["on", 20], ["off", 80]]
      }
    }
  }
}
```

flagd 监听 ConfigMap 文件变更，应用通过 OpenFeature SDK gRPC 查询，延迟 < 1s。

### 6.4 Feature Flag 与热加载的关系

```
配置热加载（本 spec）
      │
      └──► 可直接复用 ConfigMap Watch 机制
                │
                └──► Feature Flag = 特殊类型的配置
                      （建议用 OpenFeature 统一抽象，
                        而不是直接读 ConfigMap flag 值）
```

Feature flag 演进路径：
1. **阶段 1**：直接在 `AppConfig` 中加 `boolean featureXEnabled` → 方案 A 热加载
2. **阶段 2**：引入 OpenFeature SDK + flagd sidecar，后端仍是 ConfigMap
3. **阶段 3**：替换 flagd 后端为 Flagsmith/Unleash（改一行配置，应用代码不变）

## 7. 实验结论预期与 Go/No-Go 标准

| 方案 | Go 标准 | No-Go 条件 |
|------|--------|-----------|
| 方案 A（推荐） | 延迟 < 60s，无错误，无内存泄漏 | 任一指标不达标 |
| 方案 B | 触发成功率 100%，无需 spring-cloud-context 依赖 | 需要引入 spring-cloud-context 才能工作 |
| 方案 E（基线） | 滚动更新 < 3min，0 错误率 | > 5min 或 > 0.1% 错误率 |

**最终建议**：  
- 方案 A 作为热加载实现  
- 方案 E 作为代码/结构变更的标准路径  
- 两者并行存在，通过配置类型决定走哪条路

## 9. 行业实践对照（2026 现状）

### 9.1 大背景：Spring Cloud Netflix 已成历史

- 2023 年起，Spring Cloud 团队官方停止维护 Netflix 集成（Eureka/Zuul/Ribbon/Hystrix）
- Spring Cloud 2025.1.0 "Oakwood"（2025-11 发布）继续推进非 Netflix 路线
- 替代映射已成定论：
  - Eureka → **Consul / Nacos / K8s Service**
  - Zuul → **Spring Cloud Gateway**
  - Ribbon → **Spring Cloud LoadBalancer**
  - Hystrix → **Resilience4j**
- 本 spec 排除 Netflix 套件与主流方向一致，无技术债务风险

### 9.2 主流方案分布

| 方案 | 谁在用 | 本质 | 与本 spec 关系 |
|------|--------|------|---------------|
| **Spring Cloud Kubernetes Configuration Watcher** | Spring 生态 K8s 用户、大会推荐 | 独立 controller pod 监听 ConfigMap，POST `/actuator/refresh` 到应用 | **本 spec 方案 A**（推荐主线） |
| **Spring Cloud Kubernetes 应用内 Reload** | 老文章/老项目 | 应用内 watch K8s API 或 polling | **官方 deprecated since 2020**，本 spec 方案 B（不推荐） |
| **Stakater Reloader** | 中小企业、绝大多数 K8s 用户 | 控制器监听 ConfigMap/Secret SHA 变化，触发**滚动重启** | 本 spec 方案 F，作为基线对照 |
| **OpenFeature + flagd sidecar** | 走 CNCF 路线的现代团队 | gRPC sidecar，flag 评估外置 | 本 spec § 6 推荐路径 |
| **Unleash / Flagsmith / GO Feature Flag** | 需要 UI 和管理面板的团队 | 全功能 flag 管理平台，自托管 | 本 spec § 6 阶段 3 |
| **Uber Flipr / UCDP** | Uber 内部、超大规模场景 | 自建统一配置分发平台 | 量级不匹配，不适合参考 |
| **HashiCorp Consul KV** | 多语言/混合云团队 | 分布式 KV，应用 SDK 监听 | 与 spring-cloud-consul 配合也可用 |

### 9.3 关键洞察：业界"热加载"的真相

**业界绝大多数所谓的"ConfigMap 热加载"，实际是 Reloader 触发的滚动重启**，不是 in-process reload。原因：

1. **可审计**：每次配置变更对应一次 Deployment 版本，`kubectl rollout history` 可查
2. **GitOps 友好**：ArgoCD/Flux 不会因为运行时状态偏离 git 而告警
3. **状态干净**：不存在 in-flight 请求看到混合配置的边界 case
4. **零代码**：业务代码不感知，加个 annotation 就行

**Reloader 的两种策略（2026 状态）**：
- **Env Var（默认）**：往 Pod 模板注入 `STAKATER_*_RELOADER_<sha>` 环境变量。简单，但与 ArgoCD 配合时会被检测为"drift"
- **Annotation（GitOps 推荐）**：只改 pod template annotation，触发滚动但不改环境变量，ArgoCD/Flux 不报警。**2026 年新部署强烈建议这个**

### 9.4 真正需要 in-process 热加载的场景（业界共识）

只有 3 类场景值得绕过 Reloader 走 in-process reload：

1. **日志级别动态调整**：排查 prod 问题时，不希望重启丢上下文 → Spring Boot Actuator `loggers` 端点足够，无需 ConfigMap watch
2. **Feature Flag**：变更频繁、需要按用户/百分比定向 → 应该走 OpenFeature，**不要**自己用 ConfigMap 实现
3. **极高 SLA 服务的限流/超时调整**：滚动重启的几十秒不可接受 → 这才是本 spec 方案 A 的真正价值场景

### 9.5 OpenFeature 生态 2026 进展

- **OFREP**（OpenFeature Remote Evaluation Protocol）目标 Q2 2026 GA
- **OpenFeature Operator**：K8s CRD `FeatureFlag`，自动注入 flagd sidecar，无需手写 deployment YAML
- **OpenFeature MCP server**（KubeCon EU 2026 公布）：可让 AI agent 安全操作 flag
- **GitHub Action for flag cleanup**：自动检测代码中废弃的 flag
- 主流后端（Flagsmith, Unleash, GO Feature Flag, LaunchDarkly, DevCycle）都已实现 OpenFeature provider，**应用代码 vendor-neutral**

### 9.6 决策建议（更新版）

```
变更类型？
├── Feature Flag → OpenFeature + flagd（不要碰 ConfigMap）
├── 日志级别 → Actuator /loggers 端点（不要碰 ConfigMap）
├── 高 SLA 关键参数（限流/超时） → 本 spec 方案 A（Configuration Watcher + @RefreshScope）
└── 其他所有配置 → 本 spec 方案 F（Stakater Reloader + annotation 策略 + GitOps）
```

## 11. 近 2 年 Java 大会推荐解法（关键对照）

### 11.1 Spring I/O 2025 — Joris Kuipers, "Panta Rhei: Runtime configuration updates with Spring Boot"

**最直接相关的 Java 大会演讲**（Trifork CTO, 2025-05 Barcelona, [slides](https://2025.springio.net/slides/panta-rhei-runtime-configuration-updates-with-spring-boot-springio25.pdf), [video](https://www.youtube.com/watch?v=5EQzjkvLQP8), [demo repo](https://github.com/jkuipers/spring-reload-demos))

**Joris 的核心结论（按 slide 顺序）**：

| 主题 | 推荐 |
|------|------|
| 触发刷新的 HTTP 端点 | `POST /actuator/env`（更新单个属性 + rebind ConfigurationProperties + 改 log level）；`POST /actuator/refresh`（重新加载所有 PropertySource + 刷新 @RefreshScope beans） |
| 实现机制 | `EnvironmentChangeEvent` listener 自动 rebind `@ConfigurationProperties`；`@RefreshScope` bean 是 lazy-loading proxy，刷新时整个 target 被替换 |
| K8s 上的自动触发 | **Spring Cloud Kubernetes 内置的 PropertySource Reload 已 deprecated since 2020**，不要用；推荐用独立的 **Spring Cloud Kubernetes Configuration Watcher**（一个独立 controller pod，监听 ConfigMap/Secret 变更后通过 HTTP 或 Spring Cloud Bus 通知应用） |
| Spring Cloud Bus | 官方仍存在，但 Joris 演讲现场提问"有人在用吗？"几乎无人举手——**实际已被冷落** |
| Bootstrap context | "Mostly obsolete, replaced with Spring Boot config loading"——网上很多老教程仍在讲，已过时 |
| 健康检查 | Spring Cloud Context 提供 `Refresh` HealthIndicator：bind 失败时报 DOWN。**警告**：与 readiness probe 联动可能造成 crash loop——只用作 alert，不要让 K8s 据此重启 |
| 测试要求 | 必须测试 reload 路径：考虑 binding error、partial update；用 `/actuator/env` 和 `/actuator/configprops` 验证生效结果 |

**🚨 对本 spec 至关重要的两个 Gotcha**：

1. **AOT & native images（GraalVM）不支持热刷新**——如果未来要走 Spring Boot 3.x native 路线，所有基于 `@RefreshScope` / `EnvironmentChangeEvent` 的方案都失效
2. **不要用 Hikari 重新加载 DataSource**——会出现连接泄漏；要用 dedicated 实现（如 AWS RDS 专用支持）

**对本 spec 的影响**：
- 本 spec 的约束只排除 **Spring Cloud Config Server** 和 **Netflix 套件**，**不**排除 `spring-cloud-context` 和 `spring-cloud-kubernetes`
- 因此 Joris 的整套推荐（`@RefreshScope` + `EnvironmentChangeEvent` + Configuration Watcher）**直接适用**，已落地为本 spec § 3 方案 A
- 全套要点已吸收：用 ConfigurationProperties 不读字段、加 `Refresh` HealthIndicator 仅作 alert、必测 binding error 路径、不要让 Hikari 重新加载 DataSource

## 12. 2026 新增方向（之前漏掉的关键选项）

之前的方案 A-F 集中在"hot reload"和"rolling update"二元对立。**2026 年出现了第三条路径**：把"重启"做到极快，直接消除"避免重启"这个动机。

### 12.1 CRaC（Coordinated Restore at Checkpoint）—— 最大的 2026 盲点

**核心机制**：基于 Linux CRIU 的 JVM 检查点/恢复。Spring Framework 通过 `-Dspring.context.checkpoint=onRefresh` 在 `LifecycleProcessor.onRefresh` 阶段自动 checkpoint：所有非 lazy 单例已经实例化，但 lifecycle 尚未启动。

**对本 spec 的颠覆性影响**：

| 维度 | 传统 rolling restart | CRaC restore |
|------|--------------------|--------------|
| 单 pod 启动时间 | 5-30s（Spring Boot 冷启动）| **~50ms**（PetClinic 实测 100x 加速） |
| 端到端 ConfigMap → 全部 pod 应用 | 1-3 分钟 | **5-10 秒** |
| Bean 状态干净度 | 完全干净 | 完全干净（不像热加载有 partial-update 风险） |
| 与 GraalVM 的关系 | 互斥 | **互斥**（CRaC 是 HotSpot only），但 Leyden AOT 兼容 |
| 平台限制 | 全平台 | **Linux only**（kind 自带 Linux container） |

**与 Spring Cloud Context 的天然集成**：Spring Cloud 提供 `RefreshScopeLifecycle`，在 CRaC restore 时自动触发——配合 `spring.cloud.refresh.extra-refreshables` 配置，连 Hikari DataSource 这种"不该用 @RefreshScope"的 bean 都能在 restore 时被重建（参考 [OlgaMaciaszek/crac-db-refresh-scope](https://github.com/OlgaMaciaszek/crac-db-refresh-scope) demo）。

**新方案 G**：CRaC checkpoint + Stakater Reloader 滚动 + Spring Cloud Refresh
1. 镜像构建时已经做好 checkpoint（init container 或两阶段镜像）
2. ConfigMap 变更 → Reloader 触发滚动
3. 新 pod 直接 restore，~50ms 启动 + RefreshScopeLifecycle 自动重建受影响 bean
4. **结果**：拿到了"in-process 热加载"的延迟，又拿到了"滚动更新"的状态干净 + 可审计

**何时方案 G 优于方案 A**：
- 团队不希望在业务代码中引入 `@RefreshScope` 的复杂度
- 需要刷新的 bean 不容易做成 lazy proxy（如连接池、客户端）
- 对 Linux-only 限制可接受

**何时仍选方案 A**：
- 副本数极大，每次配置变更触发全量滚动代价高
- 需要"个别 pod 单独刷新"的能力（如金丝雀验证新配置）

### 12.2 Project Leyden AOT Cache（JEP 515 in JDK 25）

OpenJDK 自家的 AOT，不是 GraalVM。**不引入 closed-world 假设**，因此 `@RefreshScope` 和 `EnvironmentChangeEvent` 仍然能用。

- JDK 25 开始 stable
- Spring Boot 3.3+ 已支持
- 启动加速 3-5x（不是 GraalVM 的 60x，但保留所有动态特性）
- 训练-记录-缓存三阶段：`-XX:AOTMode=record` → `-XX:AOTMode=create` → `-XX:AOTCache=app.aot`

**对本 spec 的影响**：作为方案 A 和方案 G 的**通用加速器**，没有取舍——直接打开。建议在 § 4.3 验证指标中补一条"启用 Leyden AOT 后冷启动延迟"。

### 12.3 OpenFeature flagd 的 in-process / sidecarless 模式

之前 § 6.3 推荐的是 flagd sidecar（gRPC RPC 模式），每个 pod 一个 sidecar。**2026 年的新模式**：

- **In-process provider**：应用直接 link flagd 的 evaluation engine，flag 评估在 JVM 内完成，无网络 I/O
- **Centralized flagd**：单个集群级 flagd 实例，通过 sync 协议把规则推到所有 in-process providers
- **资源节省**：副本数大时省下 N 个 sidecar pod
- **延迟更低**：无 gRPC 跨 socket 调用

**Java provider 0.13.0 已支持** in-process mode（[文档](https://flagd.dev/providers/java/)）。建议在 § 6 阶段 2 直接选 in-process 模式，跳过 sidecar 中间形态。

### 12.4 Dapr Configuration Building Block

跨语言、与 Spring 解耦的备选路径：

- Dapr 提供统一的 Configuration API，应用通过 sidecar 订阅配置变化
- 后端可选 Redis、PostgreSQL、Azure App Configuration 等
- **Java Spring Boot 集成（dapr-spring-boot-starter）目前仍处于 alpha**，不建议作为生产首选

**适用场景**：多语言团队，需要 Java 服务和 Go/Python 服务用同一套配置订阅协议。本 spec 默认不采用，但作为已知备选记录在此。

### 12.5 Kubernetes 1.35 In-Place Pod Resize（GA, 2025-12）

**澄清**：这是 CPU/Memory 的原地调整，**不是配置热加载**。但相关：
- 需要 Java 25 配合（部分 JVM 参数运行时可调）
- 不替代本 spec 任何方案，仅在"调资源不重启"场景下相关

### 12.6 Argo Rollouts 渐进式发布

与本 spec 互补而非替代：

- 配置变更触发滚动时，用 Argo Rollouts 做金丝雀发布（20% → 50% → 100%）
- 配合 Prometheus 指标自动 promote/rollback
- **场景**：当配置变更可能影响业务行为（如限流阈值）时，比一次性全量滚动更安全

建议在 § 4.4 加 TC-07：用 Argo Rollouts 跑配置变更的金丝雀流程，记录 rollback 触发条件。

### 12.7 Spring Boot 4.0 + Spring Framework 7（2025-11 GA）

约束指定 SB 3.5，但 4.0 已 GA。**建议同步跑 4.0 兼容性验证**：

- `BeanRegistrar` 提供运行时编程式 bean 注册——可作为方案 D（自实现）的替代实现路径
- Spring Data AOT 进一步加速启动
- Native image production-ready，sub-100ms 启动——但 `@RefreshScope` / refresh 在 native 下**仍然不可用**
- 模块化 jar 让 spring-cloud-context 不需要的部分可剥离

**对本 spec 的影响**：所有方案 A-G 在 SB 4.0 下逻辑不变，但建议建立 SB 3.5 → 4.0 升级 ADR。

### 12.8 修订后的决策树

```
配置变更类型？
├── Feature Flag（频繁、需定向） → OpenFeature in-process（方案推荐升级版）
├── 日志级别 → Actuator /loggers
├── 高 SLA 关键参数 → 三选一：
│   ├── 方案 A（Configuration Watcher + @RefreshScope）—— 个别 pod 可单独刷新
│   ├── 方案 G（CRaC + Reloader）—— 50ms 滚动 + 状态干净 + 简单代码 ⭐ 2026 新优解
│   └── 方案 F（普通 Reloader 滚动）—— 副本极少或不便引入 CRaC 时
└── 普通配置 → 方案 F（Reloader）+ Leyden AOT 加速冷启
```

**优先级建议（2026-04 实施视角）**：
1. **MVP**：方案 A + 方案 F + OpenFeature in-process
2. **进阶**：方案 G（CRaC）替换方案 A 作为热加载主路径
3. **观察项**：Leyden AOT、Spring Boot 4 升级、GraalVM native 兼容性
4. **暂缓**：Dapr、Argo Rollouts（视团队规模决定）

## 13. Demo 设计：API Gateway + 微服务 + OpenFeature header 传递

### 13.1 目标

最小可演示的端到端 demo，覆盖本 spec § 6（feature flag）核心场景：

1. API Gateway 在请求入口评估所有 feature flag
2. Gateway 把评估结果作为 `X-FF-*` header 注入到下游请求
3. 后端微服务**不引入** OpenFeature SDK，只读 header 响应——单一评估源
4. 修改 flagd ConfigMap → flagd 自动 reload → 下次评估即生效（演示热加载链路）
5. 全栈虚拟线程，无 WebFlux

### 13.2 组件清单

| 组件 | 端口 | 技术栈 | 职责 |
|------|------|--------|------|
| `gateway` | 8080 | Spring Cloud Gateway **MVC** + OpenFeature Java SDK + flagd provider（RPC） | 路由 + flag 评估 + header 注入 |
| `order-service` | 8081 | Spring Boot MVC + virtual threads | 处理 `/orders/**`，根据 `X-FF-Order-Tier` 选择处理路径 |
| `pricing-service` | 8082 | Spring Boot MVC + virtual threads | 处理 `/pricing/**`，根据 `X-FF-New-Pricing` 选择算法 |
| `flagd` | 8013 (gRPC) | flagd 0.13.x | flag 评估守护进程，从 ConfigMap 读规则 |

### 13.3 Flag 设计

| Flag Key | 类型 | 默认值 | 注入 Header | 说明 |
|----------|------|--------|------------|------|
| `new-pricing-algo` | boolean | `false` | `X-FF-New-Pricing` | 是否走新定价算法 |
| `order-tier` | string variant | `standard` | `X-FF-Order-Tier` | 按 `X-User-Id` targeting，返回 `standard` / `premium` |

flagd 规则放在 ConfigMap `flagd-config`，挂载为 `/etc/flagd/flags.json`，flagd 自身负责文件 watch + reload。

### 13.4 请求流程

```
GET /orders/123  Header: X-User-Id=u-vip-001
   │
   ▼
[gateway]
  ├─ OpenFeatureFilter:
  │     ctx = {targetingKey: "u-vip-001"}
  │     newPricing  = client.getBooleanValue("new-pricing-algo", false, ctx)
  │     orderTier   = client.getStringValue("order-tier", "standard", ctx)
  ├─ 注入 header:
  │     X-FF-New-Pricing: true
  │     X-FF-Order-Tier: premium
  └─ 转发 → http://order-service:8081/orders/123
                                            │
                                            ▼
                                  [order-service]
                                    读 X-FF-Order-Tier=premium
                                    返回 premium 处理结果
```
